# Phase 1 — Repository Analysis

## 1.1 How artwork works today

### Ownership

Artwork is owned entirely by **four nullable columns on `GameEntity`** (`core-data`):

| Column | Meaning | Written by |
|---|---|---|
| `artwork_uri` | full-screen XMB background | scraper, manual pick, backup restore |
| `hero_uri` | hero banner | scraper, manual pick |
| `logo_uri` | clear logo overlay | scraper, manual pick |
| `icon_uri` | horizontal grid icon | scraper (SGDB), manual pick |

Values are either **absolute internal file paths** (`{filesDir}/artwork/...`) or, occasionally, **raw remote URLs** (when a download failed but a URL was found — see `MetadataRepository.updateMetadata` fallbacks). The UI (Coil `AsyncImage` throughout `feature-xmb`) loads whatever string is in the column.

### File layout (internal storage — the only store)

```
{filesDir}/artwork/
    {gameId}/hero.jpg                    ← scraper (CardArtworkProcessor.rawFile)
    {gameId}/background.jpg
    {gameId}/logo.png
    {gameId}/icon.jpg
    {gameId}/{type}_{timestamp}.{ext}    ← manual picks (versioned names, old files never deleted)
    cards/{platformId}/{gameId}.png      ← legacy card compositing path (processBoxArt — mostly unused now)
```

**The folder key is the Room autogenerated `gameId`.** This is the root cause of non-portability: a fresh install assigns new IDs, so files can never reconnect without the exact same database.

### Scrape pipeline

- `ArtworkRepository` (feature-artwork) — batch orchestration: `fetchMissingArtwork`, `scrapeMissingOnly`, `reScrapeAllGames`, file-aware `computeStatus()` (complete/missing/stale), `clearAllArtwork()` (deletes `filesDir/artwork` wholesale).
- `MetadataRepository` — per-game source priority TheGamesDB → IGDB → SteamGridDB, per-asset winners, then **directly calls `CardArtworkProcessor.downloadRaw` with hardcoded filenames** and persists via `GameDao.updateMetadata` (COALESCE — never overwrites non-null).
- `CardArtworkProcessor` — download (via Coil, bitmap in memory), save JPEG/PNG, **owns the path scheme** (`cardFile`, `rawFile`).
- `GameDetailViewModel` (feature-xmb) — the Artwork Manager UI: per-type picks from SGDB/TGDB/IGDB grids, **its own `copyToAppStorage()`** for local SAF picks (duplicated storage logic), `versionedArtworkFileName()`.

### Scraper identity signals available

- `steamGridDbId` column on `GameEntity` (declared but **never written** — `MetadataRepository` gets an SGDB id per fetch and discards it).
- `scrapedTitle` (TGDB canonical title), `userTitleOverride`.
- No TGDB/IGDB ids are persisted anywhere.
- No ROM serial extraction exists anywhere in the codebase.

### Game identity / scanning

- Dedupe key for ROM games is the **unique index on `rom_path`** (raw filesystem path, derived even for SAF-scanned ROMs). `rom_uri` (SAF) is secondary.
- Apps are keyed by `package_name`; harvested shortcuts by `package_name` + `launch_shortcut_id`.
- `UnmatchedRomEntity` + review flow already exists for ROM↔platform matching — a precedent for a "needs review" queue.

### Coil

Single injected `ImageLoader` (`ArtworkModule`): 20% memory cache, 512 MB disk cache in `cacheDir/artwork_cache`, crossfade. Loads file paths, URLs, and `content://` URIs equally well (Coil has a built-in content-URI fetcher).

### SAF patterns already in the codebase (mature, reusable)

- `RomRootRepository` — multi-root persisted tree grants, newline-joined DataStore key, docId↔raw-path math, `persist(uri, writable)`.
- `BackupFolderRepository`, `MediaRootRepository` — same shape.
- Scanners use **`DocumentsContract` directly (not `DocumentFile`)** for performance (`SafChildren`) — the right pattern for large trees.
- Backups are written via `DocumentsContract.createDocument` + stream copy.

### Backup / Restore

`BackupManager` bundles the **entire `filesDir/artwork` tree into the backup ZIP** (`BUNDLED_FILE_ROOTS`), serializes all tables including `games` with their IDs, and on restore rewrites the `…/files/…` prefix onto the new package's filesDir. Portability today therefore only works *through a backup file*, and every backup carries the full artwork payload (easily hundreds of MB).

### Background execution

Only `BackupWorker`/`RestoreWorker` use WorkManager. **Scrapes run in `viewModelScope`** — a full-library scrape dies if the user leaves the settings screen or the process is killed. (Flagged as an improvement in Phase 3.)

## 1.2 Coupling map

| Class | Verdict |
|---|---|
| `GameEntity` / `GameDao` | **Extend** (new columns; existing artwork columns stay as the *resolved cache* the UI reads) |
| `ArtworkRepository` | **Extend/refactor** — orchestration stays; file-existence logic moves behind storage abstraction |
| `MetadataRepository` | **Refactor seam** — must stop knowing filenames/paths; asks a store to save |
| `CardArtworkProcessor` | **Split** — download/compositing stays; path ownership moves out |
| `GameDetailViewModel.copyToAppStorage` | **Replace** with the same store call (removes duplication) |
| `BackupManager` | **Extend** — conditional artwork bundling |
| `RomRootRepository` et al. | **Unchanged** — copied as a pattern for the artwork folder repository |
| Coil setup, XMB UI, scanners | **Unchanged** — UI keeps reading `Game.artworkUri` etc.; only the *values* change |

Tightly coupled today: scraper ↔ storage (hardcoded paths in three classes), identity ↔ Room PK (folder names), backup ↔ internal storage layout.

# Phase 2 — Design

## 2.1 Stable artwork identity

### Candidate approaches

| Approach | Pros | Cons |
|---|---|---|
| **A. Room ID (status quo)** | zero work | dies on reinstall; rejected by requirements |
| **B. ROM content hash (CRC32/SHA-1 of file)** | perfect identity, survives renames, matches No-Intro DATs | must read every ROM (GBs over SAF = minutes-to-hours on first index); zips/CHDs need per-format handling; useless for apps/manual entries |
| **C. ROM serial (header parse)** | robust for disc platforms (PSX/PS2/PSP/GC…) | needs per-platform parsers we don't have; meaningless for cart ROMs without DBs; big new surface |
| **D. Scraper IDs (SGDB/TGDB/IGDB)** | globally stable, cross-device | only exists after a successful scrape; manual/local-only artwork has none; ties identity to third-party services |
| **E. Platform + normalized ROM filename** | free (we already have it), human-readable, survives device moves when the user copies their ROM folder as-is, works offline, debuggable by eye | breaks if the user renames a ROM; collides if two files normalize identically on one platform |
| **F. Package name (apps)** | canonical for Android apps | apps only |

### Recommendation: **layered identity — E/F as the primary key, B and D as recorded *evidence* in metadata**

The primary key must be computable from a bare `Game` row with no I/O and no network, or the system can't address artwork before scraping. That rules out B, C, D as primaries.

**`ArtworkKey` (conceptual):**

- ROM games: `rom/{platformId}/{slug}` where `slug` = normalized filename stem — lowercase, strip extension and disc/region tags `(USA)`, `[!]`, `(Disc 1)` normalized to a `disc1` suffix, non `[a-z0-9._-]` → `-`, collapse repeats, trim to 100 chars. Deterministic and documented so the Theme Studio / other tools can compute it too.
- Apps / Windows-card games: `app/{packageName}` (+ `/{shortcutId-hash}` for harvested shortcuts).
- Manual entries with neither: `manual/{slug(title)}-{8-char-random}` minted once and persisted (the one case where the key must be stored, not derived).
- Collisions: append `-2`, `-3` at mint time (checked against the index).

**Evidence recorded in `metadata.json`** (not part of the folder name): rom filename as-seen, rom file size, optional CRC32 (computed lazily/opportunistically, never blocking), SGDB/TGDB/IGDB ids when a scrape supplied them, scraped title, platform. This is what the confidence matcher uses when the primary key misses (renamed ROM). Renames degrade gracefully to a "likely match" instead of a hard loss.

## 2.2 Storage abstraction — scraper/storage separation

New module-internal layer in `feature-artwork` (interfaces in a small `api` package so `feature-xmb` and `feature-backup` can consume them without knowing the implementation):

```
ArtworkStore                     ← the only thing scrapers & UI talk to
 ├─ save(key, kind, source)      kind = ICON|HERO|BACKGROUND|LOGO; source = bytes/temp-file
 │        → ArtworkRef (loadable URI string + relative path + checksum)
 ├─ load(key, kind) → ArtworkRef?
 ├─ delete(key, kind?)           kind omitted = whole entry
 ├─ scan() → Flow<ArtworkEntry>  streaming enumeration for verify/rebuild (100k-safe)
 ├─ writeMetadata(key, ArtworkMetadata) / readMetadata(key)
 └─ stats() → counts, bytes

ArtworkKeyFactory                pure function Game → ArtworkKey (unit-testable, also used by Studio later)

ArtworkStorageProvider           where bytes live; two implementations:
 ├─ InternalStorageProvider      filesDir/artwork2/… (fast, default, pre-migration)
 └─ SafStorageProvider           user's SAF tree via DocumentsContract (portable mode)

ArtworkMigrator                  internal ⇄ portable copy with verification + rollback
ArtworkMatcher                   reconnect engine (see 2.6)
ArtworkIndexDao (Room)           fast lookup cache: key → docId/relative path per kind
```

Rules:

- `MetadataRepository` and `GameDetailViewModel` call **only** `ArtworkStore.save(...)`. They never build a filename or path again. `CardArtworkProcessor` shrinks to "download URL → temp file/bitmap" and hands bytes to the store.
- The store returns the **string the UI should put in `Game.artworkUri`** — an absolute file path (internal mode) or a `content://` document URI (portable mode). The XMB UI is untouched; Coil loads both.
- Writes always go: temp file in `cacheDir` → verified non-empty → moved/streamed into the provider → index row upserted → metadata.json updated. No partial file is ever visible under its final name.

### Read-path performance decision (important trade-off)

Two options for portable mode:

1. **Serve straight from SAF** (`content://` URIs in the DB). Single copy of every file; Coil's 512 MB disk cache absorbs repeat decodes. Cost: first-load latency per image (binder round-trip + provider open), and a dead grant makes artwork vanish instantly.
2. **Write-through mirror**: SAF folder is the source of truth; internal `filesDir` keeps a hot copy; DB stores the internal path. Fast and grant-loss-tolerant, but doubles disk usage and adds sync complexity.

**Recommendation: option 1 (direct SAF serving) with the Room index**, for one-copy simplicity — Coil's caches make steady-state performance a non-issue on the XMB, and doubling a multi-GB library on internal storage defeats the point of putting it on an SD card. Mitigations: `verify` action + startup grant check surface dead grants explicitly (see 2.9); the index avoids all directory enumeration on the hot path. If real-device testing shows unacceptable first-scroll latency, the mirror becomes a bounded LRU *icon-only* cache (icons are small; heroes/backgrounds load once per detail view) — the abstraction supports adding this later without schema changes.

## 2.3 Folder structure

```
Play Field Portal Artwork/            ← user-picked SAF tree
  pfp-artwork-library.json            ← library manifest (format version, library UUID,
                                         createdAt, app version, entry count hint)
  games/
    {platformId}/                     ← e.g. gba/, psx/ — mirrors how users think & browse
      {slug}/                         ← one folder per game = one ArtworkKey
        icon.jpg|png
        hero.jpg|png
        background.jpg|png
        logo.png
        manual.pdf                    ← ScreenScraper (optional, toggle-gated)
        video.mp4                     ← ScreenScraper video snap (optional, toggle-gated)
        metadata.json
  apps/
    {packageName}/
      icon.png … metadata.json
  manual/
    {minted-slug}/ …
```

Why each level exists:

- **Manifest at root** — lets a fresh install recognize "this is a PFP artwork library" instantly (one document read), carries the format version for forward migration, and the library UUID distinguishes "same library re-linked" from "different library" in settings.
- **`games/{platformId}/`** — keeps any single directory small (SAF child enumeration degrades badly past a few thousand children; per-platform fan-out keeps it in the hundreds), and makes the folder human-navigable — a user can drop `cover.jpg` next to a game by hand and *Reload Library* picks it up.
- **One folder per game** — atomic unit for copy/delete/review; metadata travels with its images.
- **Fixed asset filenames per kind** — no timestamp-versioned junk (today's manual picks leak old files forever); replacing an asset overwrites one well-known name. Extension may be jpg or png; the index + metadata record which.

**Filename safety:** slugs restricted to `[a-z0-9._-]`, ≤100 chars — safe on FAT32/exFAT SD cards (case-insensitive, no reserved chars), safe as SAF display names, no Windows reserved names (`con`, `aux` → suffixed). Normalization is case-folding, so two ROMs differing only by case collide → collision suffix at mint time.

**Duplicate prevention:** the index enforces one entry per (key, kind); saves overwrite. Cross-key duplicates (same game under two platforms/names) are legitimate and left alone — a *Cleanup* report can list byte-identical files by checksum as advisory only.

**Future expansion:** additive — new asset kinds are new filenames inside the entry (e.g. `screenshot1.jpg`, `manual.pdf`); new content types are new top-level dirs (`videos/`, `collections/`); format changes bump the manifest version.

## 2.4 metadata.json (per entry)

JSON via kotlinx-serialization (already the project standard; human-readable, hand-editable, diff-able — the right call for a user-owned folder; a binary format would betray the "user-owned" premise).

Stores: `formatVersion`, `key`, `platformId`, `title`, `scrapedTitle`, identity evidence (`romFileName`, `romSize`, `romCrc32?`), scraper ids (`ssId`, `sgdbId`, `tgdbId`, `igdbId`), **descriptive metadata block** (`description`, `developer`, `publisher`, `releaseYear`, `genre`, `players?`, `rating?` — so a device migration restores text metadata with zero API calls; DB `userTitleOverride`/`userNote` always win over folder values on reconnect), per-asset records (`kind` = icon/hero/background/logo/manual/video, `fileName`, `source` = screenscraper/sgdb/tgdb/igdb/local/unknown, `sourceUrl?`, `sha256`, `savedAt`), `updatedAt`.

Versioning/migration: `formatVersion` int; readers use `ignoreUnknownKeys`; older-version files are upgraded lazily on next write; a newer major version than the app supports → entry treated as read-only and surfaced in *Verify*. Corrupted/missing metadata never blocks image loading (images are matched by well-known filenames); it only degrades matching confidence.

## 2.5 Database changes (why each exists)

All additive — one Room migration (v23 → v24):

1. `games.artwork_key TEXT` (indexed, nullable) — the stable identity, minted on first artwork save or first index pass. *Why in the DB:* joins the volatile Room id to the portable world; recomputing on every load is wasteful and manual keys can't be derived.
2. `games.tgdb_id INTEGER`, `games.igdb_id INTEGER`, `games.ss_id INTEGER` (ScreenScraper), `games.rom_crc32 TEXT` — persist scraper ids we already fetch and discard; also **start actually writing the existing `steam_grid_db_id`**. *Why:* prime matching evidence; removes re-search calls on re-scrape; `rom_crc32` is computed by the ScreenScraper hash lookup anyway and doubles as matcher stage-4 evidence.
3. **New table `artwork_index`** (`key`, `kind`, `location` = INTERNAL|PORTABLE, `docUriOrPath`, `sha256`, `sizeBytes`, `updatedAt`; PK (key, kind)) — the fast lookup cache over the folder. *Why:* the folder is the source of truth but SAF enumeration is far too slow for UI or even per-launch checks; the index is rebuildable at any time (Reload Library), so losing it is never data loss.
4. **New table `artwork_match_review`** (`key`, `candidateGameId`, `confidence`, `signalsJson`, `status`) — persistence for the Needs-Review queue (mirrors the existing `UnmatchedRomEntity` precedent). *Why:* review sessions must survive process death mid-migration.

DataStore additions (ride the existing backup key mechanism): `artwork_folder_tree_uri`, `artwork_storage_mode` (`internal` | `portable`), `artwork_library_uuid`.

Existing `artwork_uri`/`hero_uri`/`logo_uri`/`icon_uri` columns **stay** — they remain the resolved, UI-facing values (now sometimes `content://`). No UI change.

## 2.6 Matching engine (device-migration reconnect)

Runs when a folder is linked and games exist (fresh install + old library), or on *Find Missing Artwork*. Stages, per artwork entry, evaluated against the game table (all lookups via in-memory maps built once — O(entries + games)):

| Stage | Signal | Confidence |
|---|---|---|
| 1 | `ArtworkKey(game) == entry.key` | **Exact** — auto-link silently |
| 2 | scraper id match (SGDB/TGDB/IGDB id in metadata == id on a game row of same platform) | **Exact** — auto-link |
| 3 | `romFileName` + `romSize` match a game's ROM (same platform) | **Exact** — auto-link (rename-proof when user kept file) |
| 4 | CRC32 match, when both sides have one | **Exact** — auto-link |
| 5 | normalized-title match (entry title/scrapedTitle vs game displayTitle/scrapedTitle, same platform) | **Likely** — auto-link, flagged; listed in a post-run summary with one-tap undo |
| 6 | fuzzy title (trigram/Jaro-Winkler ≥ threshold, same platform) | **Needs Review** — queued, never auto-linked |
| 7 | nothing | **Unmatched** — kept in folder untouched, visible in review UI with "assign to game…" |

Linking = set `games.artwork_key`, resolve asset URIs into the four game columns, upsert index rows. The reverse direction (game with no matching entry) feeds *Find Missing Artwork* → scrape targets. Everything is idempotent and resumable (review table persists state).

## 2.7 Migration (internal → portable) & rollback

Foreground-service WorkManager job (pattern exists in `BackupWorker`; survives screen-off, shows notification progress via `BackgroundTaskNotifier`):

1. **Preflight:** grant is writable; free-space estimate vs current artwork bytes; create manifest.
2. **Copy phase (non-destructive):** for each game with artwork — mint key, write assets + metadata.json into the tree, verify each write (re-open, size check), upsert index. Internal files untouched. Resumable: index rows mark progress; re-run skips verified entries.
3. **Flip phase:** only after 100% verification — batch-update the four game columns to `content://` URIs, set `artwork_storage_mode = portable`.
4. **Grace period:** internal copies retained until user runs *Free Up Internal Space* (explicit, shows MB reclaimed) — this **is** the rollback store.
5. **Rollback:** set mode back to `internal`, batch-restore columns to internal paths (recorded in the index rows as the previous location). One transaction, instant. Available in settings while internal copies exist; after cleanup, rollback = re-migration in reverse (portable → internal copy), same machinery, direction flag.

Interrupted transfer = job re-run resumes; the flip is atomic and last, so the app is never half-migrated from the UI's point of view.

## 2.8 Backup/Restore integration

- **Portable mode:** backups **exclude** the artwork tree (the whole point — artwork already lives in a user-owned folder). The ZIP carries: `artwork_folder_tree_uri` (inert-grant pattern already used for ROM roots), storage mode, library UUID, `artwork_index` + `artwork_key`s. Restore → settings shows "Re-link artwork folder" pre-pointed at the old location → matcher stage 1 reconnects everything by key instantly.
- **Internal mode:** unchanged (tree bundled as today), plus the new tables ride along.
- Backup format version bump; older backups restore fine (no new entries → internal mode assumed).

## 2.10 Error handling & performance

| Failure | Behavior |
|---|---|
| SD card removed / grant revoked | Detected at startup (cheap: one `contentResolver.persistedUriPermissions` check + manifest stat) and on first failed load. Mode stays portable; UI shows re-link banner; Coil failures fall back to the existing PortalIcon placeholder (already the no-artwork path). Optional later: auto-fallback to stale internal copies if grace-period files remain. |
| Folder deleted but grant alive | Manifest missing → treated as "empty library"; *Reload* offers to rebuild from app state (re-export from internal copies / re-scrape). Never auto-deletes DB references without user action. |
| Corrupted metadata.json | Entry still serves images (well-known filenames); metadata rewritten from DB on next save; Verify reports it. |
| Duplicate/invalid foreign files in tree | Ignored by index (unknown names/dirs skipped), listed by Verify as "foreign files", never deleted automatically. |
| Partial import / interrupted write | Temp-file + rename discipline; index row written only after verified save; Verify sweeps leftovers. |
| 100k+ entries | No full-tree enumeration on any hot path (index only). Scan/verify are streaming (`Flow`, DocumentsContract cursor paging, no `DocumentFile` recursion), constant memory, cancellable, run on `Dispatchers.IO` with bounded parallelism (SAF providers serialize writes anyway; 2–4 concurrent copies is the sweet spot). Matching is map-based, O(n). |
| Memory | Never hold bitmaps for storage ops — stream bytes (today's `downloadRaw` decodes to Bitmap then re-encodes, losing quality and burning memory; the store streams the original bytes and keeps the original format). |
| Cancellation | All workers cooperative via coroutine cancellation; migration/copy resumable by design. |

## 2.11 ScreenScraper re-integration

### History & post-mortem

ScreenScraper was PFP's Tier-1 source until commit `d0d375c` (June 23 2026) removed it wholesale (368-line `ScreenScraperApi.kt`, credentials UI, manuals/video-snap options, diagnostics). Root cause of the failure it papered over: **`SS_DEV_PASSWORD = ""` — the developer account was never registered.** The API rejects/starves requests without valid `devid`/`devpassword`, so the integration could never have worked reliably regardless of user credentials.

### Prerequisite (blocking, non-code)

Register a developer account: create a normal screenscraper.fr account, then post in the developer forum section (`forumsujets.php?frub=12`) presenting Play Field Portal (free app — satisfies their free-software licensing requirement) and request API identifiers. The team replies with `devid`/`devpassword`. **No SS code lands until these exist.**

### Credential model

- **Dev credentials** — injected at build time from `local.properties`/CI secret → `BuildConfig` (same pattern as the signing config; never committed). Accepted reality: extractable from the APK, like every frontend that embeds them; `softname=PlayFieldPortal` identifies us server-side.
- **User credentials** (`ssid`/`sspassword`) — optional but strongly encouraged in UI (raises thread count + daily quota). Stored via `KeystoreSecretCipher` like TGDB/IGDB secrets; added to backup's `ENCRYPTED_CREDENTIAL_KEYS` (device-bound, dropped on cross-device restore).

### Client design (restore + modernize `ScreenScraperApi` from `d0d375c~1`)

Keep: response models, region-priority media selection (`us → wor → any`), `SsFailureReason` diagnostics, platform→systemId map, streamed hashing.
Modernize:

1. **Hash sources** — old code hashed `java.io.File` only. New `RomHasher` abstraction: raw path → stream; SAF-only ROM (`romUri`) → `contentResolver.openInputStream` stream; zipped cartridge ROM (DB v23 support) → hash the *inner* entry via `ZipInputStream` (single-entry policy, matching how the zip support scans). CRC32 only (MD5 adds nothing for matching and doubles I/O). Size cap: skip hashing above a threshold (default ~256 MB — disc images match fine by filename+size on SS; threshold configurable in code, not UI). Computed CRC is persisted to `games.rom_crc32` and metadata.json — shared with the portable-identity matcher.
2. **Correct request shape (fixes vs old client, per official WebAPI v2 docs):**
   - Base URL `https://api.screenscraper.fr/api2/` (old code hit `www.screenscraper.fr` — known TLS-cert mismatch reports against the www host).
   - **Send `romtaille` (file size)** — the documented matching tuple is hash + size + filename; the old client never sent size, weakening every non-hash lookup. Size is free (`File.length()` / SAF cursor) even when hashing is skipped.
   - **Send `gameid` on re-scrape** — once `games.ss_id` is persisted (M2), refreshes skip matching entirely and can't mismatch.
   - `serialnum` parameter exists for disc serials — not used at launch (no serial extraction in PFP), but the request layer models it so a future serial reader plugs in.
3. **Request/limits hygiene** — parse the `ssuser` block in responses (`maxthreads`, `requeststoday`, `maxrequestsperday`); serialize SS calls behind a per-source rate gate (1.1 s pacing, single-flight; other sources keep 500 ms). Map the documented HTTP codes to typed `SsFailureReason`s: 400 malformed, 401 API closed to non-members, 403 bad dev credentials, 404 no match, 429 thread/minute limit, **430 daily quota exceeded, 431 too-many-unrecognized-ROMs** — the old client only knew 401/403/429. Batch behavior: 429 → wait-and-retry once; 430 → abort SS for the rest of the run (other sources continue); 431 → stop submitting *unhashed* lookups for the run (hash lookups may continue). Handle plain-text error bodies without JSON parse crashes.
4. **New assets** — manuals (`manuel` media → `manual.pdf`) and video snaps (`video-normalized` → `video.mp4`) as new `ArtworkKind`s saved through `ArtworkStore` like any other asset; both behind restored scrape-option toggles (default OFF — they're large). The Game Detail "Manual" action (currently a stub message) opens the stored PDF via an `ACTION_VIEW` intent (FileProvider/content URI); video snap playback deferred to a later milestone (stored but unused is fine).

### Pipeline position

Restore hash-first priority: **ScreenScraper (hash, then filename) → TheGamesDB → IGDB → SteamGridDB (artwork-only)**. SS returns in one call what currently takes three; on an SS hit the other sources only fill asset gaps. `updateMetadata` re-gains `developer`/`publisher`/`genre` (dead columns since the removal). SS id joins the persisted scraper-id set; SS becomes a fourth source in the Game Detail Artwork Manager source row (`SOURCE_SS`).

### Settings UI additions

ScreenScraper account section (username, password, Test — calls `ssuserInfos.php` and shows thread/quota info, Clear), source-priority text updated, Manuals/Video-Snaps toggles restored, per-run diagnostics surfaced in scrape progress (source names already flow through `ScrapeProgress`).

# Phase 3 — Architecture Review (self-critique)

1. **Overall:** clean seam (store interface) at exactly the point of today's worst coupling; folder is source of truth, Room is a rebuildable cache — the only arrangement where "reconnect on a new phone" is trivially correct.
2. **Risks:**
   - *SAF write throughput* is the #1 risk. Per-document create/open costs ~5–20 ms binder overhead; a 5,000-entry migration ≈ 20k documents ≈ minutes, not seconds. Mitigated by resumable worker + honest progress UI; must be measured on-device early (Milestone 3 gate).
   - *Key drift*: if slug normalization ever changes, existing folders stop matching at stage 1. Mitigation: normalization spec is versioned in the manifest and frozen after v1; changes require a matcher stage, not a rewrite.
   - *Two sources of truth during grace period* (internal + portable). Mitigated by making the flip atomic and internal copies read-only leftovers, never served in portable mode.
3. **Weaknesses:** filename-based identity is genuinely weaker than content hashing — a user who renames ROMs *and* has no scrape ids lands in fuzzy matching. Accepted: opportunistic CRC32 backfill (hash small ROMs during idle scans) closes most of this over time without ever blocking.
4. **Scalability:** index-first design holds to 100k+; the one sub-linear-hostile spot is SAF child listing during full rescan — bounded by per-platform fan-out and cursor streaming.
5. **Future-proofing:** asset kinds, content types, and metadata are all additive; the manifest version gates format evolution; the same store abstraction can later host video thumbnails or theme assets.
6. **Performance:** read path unchanged in internal mode; portable mode adds first-load content-resolver cost only (Coil disk cache thereafter). Migration/verify are the only heavy ops and are workers.
7. **Android pitfalls:** persistable-grant cap (512 on modern Android — we add 1); FAT32 filename limits (handled by slug rules); `DocumentFile` recursion trap (avoided — DocumentsContract only, per existing scanners); process death mid-write (temp+rename); scoped-storage direct-path access to SD cards is not assumed anywhere.
8. **SAF limitations:** no atomic rename-over-existing on all providers (delete-then-create with index updated last); no cheap recursive listing (design never needs one on hot paths); `lastModified` unreliable on some providers (we use our own `updatedAt`).
9. **Room migration risk:** low — purely additive columns/tables. The real risk is the *data* backfill (minting keys for existing rows), which is deliberately **not** in the SQL migration but a lazy/startup pass — a crash mid-backfill just resumes.
10. **Improvements over my own first instinct:** I initially considered mirroring everything internally (write-through cache). Rejected: doubles storage, adds a sync protocol, and mostly defends against a failure (dead grant) that's better handled by explicit UI. Also: move batch scrapes to WorkManager while we're touching this path — cheap win, real user-facing reliability gain.

# Phase 4 — Implementation Roadmap

> Each milestone ships independently and leaves the app fully working. No milestone starts until the previous one is verified.

### M1 — Storage seam (pure refactor, no behavior change)
- **Goal:** `ArtworkStore` + `InternalStorageProvider` wrapping today's exact layout; `MetadataRepository`, `GameDetailViewModel`, `ArtworkRepository` stop touching paths; byte-streaming replaces bitmap re-encode in `downloadRaw`.
- **Files:** feature-artwork (new `store/` package; `MetadataRepository`, `CardArtworkProcessor`, `ArtworkRepository`), feature-xmb (`GameDetailViewModel`).
- **Risks:** silent path regressions. **Testing:** unit tests for store contract; manual scrape + manual pick + clear on device; existing `ArtworkSettingsViewModelTest` green. **Rollback:** git revert — no schema/data changes. **Deps:** none.

### M2 — Stable identity + metadata + index (still internal-only)
- **Goal:** `ArtworkKeyFactory` (+ exhaustive unit tests for slug rules), Room v24 migration (columns + `artwork_index` + review table), scraper ids persisted, metadata.json written on every save, lazy key backfill.
- **Files:** core-data (entities, DAOs, `PFPDatabase`), feature-artwork.
- **Risks:** migration correctness; key collisions. **Testing:** Room migration test (v23→v24 with real schema JSONs), key-factory table-driven tests, backfill idempotence test. **Rollback:** columns are nullable/ignored if later milestones never ship. **Deps:** M1.

### M2.5 — ScreenScraper integration (on the new seam)
- **Goal:** dev credentials wired via BuildConfig; `ScreenScraperApi` restored + modernized (`RomHasher` with raw/SAF/zip-inner sources, typed error handling, rate gate, quota awareness); pipeline reordered SS-first; `developer`/`publisher`/`genre` persisted again; SS id + CRC32 recorded; manuals/video snaps as new `ArtworkKind`s (toggles default OFF); Manual action opens stored PDF; SS credentials + test UI in Artwork Settings; `SOURCE_SS` tab in Artwork Manager.
- **Files:** feature-artwork (`ScreenScraperApi`, `RomHasher`, `MetadataRepository`, `ScrapeOptions`, `ArtworkScrapePreferences`, `MetadataApiKeyProvider`), feature-settings (screen + VM), feature-xmb (`GameDetailViewModel` source row, Manual action), feature-backup (credential key list), core-data (none beyond M2's columns).
- **Risks:** dev-account approval latency (external); SS server availability/quota behavior under batch scrapes; zip-inner hashing edge cases (multi-entry zips → fall back to filename matching). **Testing:** unit tests for `RomHasher` (file/SAF/zip fixtures) and error-body parsing; live smoke test against SS with real credentials (small batch, verify quota headers respected); regression run of TGDB/IGDB-only path with SS creds absent. **Rollback:** SS is additive at the head of the chain — absent credentials short-circuits to today's behavior exactly. **Deps:** M1 (store seam), M2 (id/CRC columns), **external: approved dev credentials**.
- Note: hashing during a scrape is the first place large-file SAF reads happen — its measurements feed M3's perf gate.

### M3 — SAF provider + folder picker (opt-in, new-artwork-only)
- **Goal:** `SafStorageProvider` (DocumentsContract, temp+verify writes), `ArtworkFolderRepository` (clone of `BackupFolderRepository`, writable grant), settings Storage row + picker + manifest detection; new saves land portable; **on-device perf measurement gate** (write throughput, first-load latency on XMB).
- **Files:** core-data (folder repo), feature-artwork (provider), feature-settings (screen + VM).
- **Risks:** provider quirks (SD vs internal vs cloud DocumentsProviders); perf. **Testing:** instrumented tests against the platform ExternalStorageProvider; manual on SD card + USB drive. **Rollback:** feature flag / storage-mode stays `internal` by default. **Deps:** M2.

### M4 — Migration worker + rollback + free-up
- **Goal:** `ArtworkMigrator` worker (resumable copy → verify → atomic flip), rollback action, *Free Up Internal Space*, grant-loss detection + re-link banner.
- **Files:** feature-artwork (migrator, worker), feature-settings, core-ui (notifier reuse).
- **Risks:** interruption mid-copy; disk-full. **Testing:** kill-process-mid-migration instrumented test; disk-full simulation; large-library timing on device. **Rollback:** built-in (grace-period copies + mode flip). **Deps:** M3.

### M5 — Matching engine + reconnect + review UI
- **Goal:** `ArtworkMatcher` stages 1–7, "Found existing library" reconnect flow on folder link, Review Unmatched screen, *Reload Library*.
- **Files:** feature-artwork (matcher), feature-settings (review UI), core-data (review DAO).
- **Risks:** false auto-links at stage 5. **Testing:** table-driven matcher tests (rename/retag/case scenarios); end-to-end: wipe app data, re-link folder, assert full reconnect. **Rollback:** matcher only writes `artwork_key` + resolved URIs — *Clear links* action restores pre-match state. **Deps:** M2 (M3 for the real flow; matcher itself is testable against internal store).
- **This is the milestone that delivers the headline feature.**

### M6 — Backup integration
- **Goal:** portable mode excludes artwork tree from ZIP, carries folder URI + index + keys; restore pre-points re-link; backup format version bump; restore of old backups unchanged.
- **Files:** feature-backup (`BackupManager`, manifest), feature-settings (restore hint UI).
- **Risks:** cross-version restore matrix. **Testing:** extend `BackupManagerTest` for both modes × both directions. **Rollback:** format version gate. **Deps:** M4, M5.

### M7 — Maintenance & polish
- **Goal:** Verify Library, Cleanup (orphans/foreign/dupes report), Move Artwork (folder→folder), *Find Missing Artwork* moved to WorkManager, opportunistic CRC32 backfill.
- **Files:** feature-artwork, feature-settings.
- **Risks:** low. **Testing:** verify/cleanup against deliberately corrupted trees. **Rollback:** all actions non-destructive by default. **Deps:** M4, M5.
